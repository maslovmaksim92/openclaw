### Принцип работы

Система работает **полностью автоматически, 24/7**:

1. **Program 0 (Radar)** ищет новые бизнес-сигналы через веб-поиск каждый час
2. **Program 1 (Intake)** принимает сигналы, проверяет на дубликаты и отсекает мусор
3. **Programs 4→5→6→7** — конвейер валидации: спрос → экономика → финансы → вердикт
4. **Program 7** отправляет вердикт в Telegram (topic 267 или 270)
5. **tg_notifier.py** — внешний скрипт-нотификатор, запускается отдельно через cron

## 2. Правила критичности

Применяются ко **всем программам** без исключений.

### LTV Gate (фильтр доходности)

```
МИНИМАЛЬНЫЙ ПОРОГ: LTV > 500 000 руб/мес (= 6 000 000 руб/год)
```

Как достичь порога:
- Один крупный B2B контракт: разовый ≥ 500 000 руб/мес при avg-сроке 6 мес
- Recurring SaaS: подписка ≥ 500 000 руб/мес × 12 = 6М руб/год
- Агентство/портфель: 10+ клиентов × 50 000+ руб/мес каждый

Модели, не прошедшие LTV Gate, **немедленно** отклоняются на любом этапе.

### Порог оценки

| Диапазон | Статус | Куда |
|----------|--------|------|
| ≥ 70/100 | **APPROVED** | `pipeline/approved/` + Topic 267 |
| 65–69/100 | **NEAR-PASS** | `pipeline/rejected/` + Topic 270 (особый формат) |
| < 65/100 | **REJECTED** | `pipeline/rejected/` + Topic 270 |

Статус **"on-hold" запрещён**. Каждый кейс получает финальный вердикт.

### Запрет шаблонного содержимого

Файлы кейса с заглушками (`"Что уже понятно по экономике"`, `"<case-slug>"`, пустые списки `"1. \n2. "`) — **недействительны**. Агент обязан удалить шаблонный файл и пересоздать с реальными данными.

### Язык и валюта

- Все файлы и Telegram-сообщения: **только русский**
- Все расчёты: **только рубли**. USD/EUR → конвертировать по ~90 руб/$. В скобках можно указать исходную валюту.

## 3. Program 0 — Radar & Search

**Cron:** `models-radar-*` (несколько Job-ов по секторам)  
**Задача:** найти ровно 1 новый сырой сигнал за цикл.

### Активные Radar-Job'ы

| Код | Тематика |
|-----|----------|
| `radar-b2b-ops` | AI-операции, автоматизация B2B |
| `radar-ai-services` | AI-услуги на заказ |
| `radar-quick-ai` | Быстрые AI-сервисы (боты, фото, песни) |
| `radar-geo-expansion` | Западные модели без российского аналога |
| `radar-healthcare` | AI в медицине |
| `radar-fintech` | AI в финансах |

### Что НЕ записывать (LTV Gate для Radar)

- Фриланс и разовые проекты
- Инфопродукты с малым чеком
- B2C SaaS дешевле 5 000 руб/мес без масштаба
- Вендорские PR-анонсы без независимой возможности

### Алгоритм

```
1. Проверить pipeline/raw — если есть необработанный файл → NO_REPLY
2. Найти 1 новый сигнал через веб-поиск
3. LTV Gate: потенциал > 500 000 руб/мес? Нет → пропустить, искать следующий
4. Обязательно проверить существующие продукты в РФ:
   - Поиск Telegram-ботов: @keyword_bot, @keywordai_bot, @ai_keyword_bot
   - curl "http://172.17.0.1:9001/telegram-search?keyword=СЛОВО"
   - App Store / Google Play Russia
   - vc.ru, Habr, tgstat.ru
5. Записать raw-файл в pipeline/raw/
6. НЕ открывать кейс, НЕ делать handoff
```

### Формат raw-файла

```yaml
name: [название модели]
source: [URL + дата]
sector: [B2B-OPS | AI-SERVICES | QUICK-AI | HEALTHCARE | FINTECH | GEO-EXPAND]
ltv_signal: [первичная оценка потенциала LTV в рублях]
existing_products_ru: [список аналогов с числами подписчиков]
why_it_may_matter:
  - [причина 1 с цифрами]
  - [причина 2 с цифрами]
  - [причина 3 с цифрами]
```

## 4. Program 1 — Intake & Triage

**Trigger:** новый файл в `pipeline/raw`  
**Задача:** отсеять мусор, проверить дубли, создать кейс или обогатить существующий.

### DEDUP CHECK — обязательно первым делом

```
Перед любым действием просмотреть ВСЕ директории:
  pipeline/cases/  +  pipeline/approved/  +  pipeline/rejected/

ДУБЛЬ = совпадают 2 из 3 критериев:
  1. Ниша / проблема  (AI поддержка клиентов ≈ AI колл-центр)
  2. Целевая аудитория (B2B enterprise ≈ B2B SaaS)
  3. Ключевая технология (LLM + голос ≈ LLM + чат)

Если дубль → ОБЯЗАТЕЛЬНО обогащать 01-evidence.md
             → ЗАПРЕЩЕНО создавать новый кейс
При сомнении → всегда обогащение, никогда создание.
Один slug = одна концепция навсегда.
```

### Алгоритм (новый сигнал)

```
1. Взять файл из pipeline/raw
2. DEDUP CHECK (см. выше)
3. Если шум → удалить, не создавать кейс
4. LTV Gate: потенциал > 500 000 руб/мес?
   Нет → pipeline/triage с пометкой "rejected: ltv-too-low"
5. Если уникальная концепция + LTV высокий →
   создать pipeline/cases/<case-slug>/00-brief.md
```

### Алгоритм (автообогащение — если дубль)

```
6. Найти совпадающий кейс в pipeline/cases/
7. Прочитать / создать pipeline/cases/<slug>/01-evidence.md
8. Добавить запись:
   ## [Дата] — [Источник]
   - Тип: поддерживающий сигнал / тренд / кросс-вендорное подтверждение
   - Источник: [URL]
   - Связь с кейсом: [почему этот сигнал укрепляет кейс]
   - Ключевые данные: [факты, числа, цитаты]
9. Переместить raw-файл в pipeline/raw/processed/
10. Записать triage-отчёт в pipeline/triage/
```

### Формат triage-отчёта

```markdown
# Триаж: [название]
## Дата: [дата]
## Решение: [новый кейс | поддерживающий сигнал | шум | дубль | ltv-too-low]
## Источник: [файл из pipeline/raw]
### Почему:
- [причина]
### Действие:
- [что сделано]
### Следующий шаг:
- [что делать дальше]
```

## 5. Program 2 — Case Validation

**Задача:** проверять модель блок за блоком, обновлять файлы кейса.


## 7. Program 4 — Demand Validation

**Cron:** `models-demand-validation`  
**Trigger:** кейс с `00-brief.md`, без `02-demand.md`  
**Задача:** глубокая мульти-сигнальная валидация спроса.

### Инструменты (без токена)

| Источник | Endpoint / метод | Что даёт |
|----------|-----------------|----------|
| Wordstat + многое | `http://172.17.0.1:9001/multi-demand?keyword=X` | Составной score |
| Google Trends RU | `http://172.17.0.1:9001/trends?keyword=X` | Индекс 0–100 |
| Telegram-каналы | `http://172.17.0.1:9001/telegram-search?keyword=X` | Подписчики |
| HH.ru вакансии | hh.ru/search/vacancy?text=X&area=113 | Точное кол-во |
| Avito объявления | avito.ru/rossiya?q=X | Кол-во объявлений |
| vc.ru + Habr | site:vc.ru X | Кол-во публикаций |

### Composite Demand Score

| Уровень | Критерии |
|---------|---------|
| **HIGH** | TG 50k+ ИЛИ Wordstat 5000+ ИЛИ (HH>200 + Avito>100 + vc>10) |
| **MEDIUM** | TG 5k–50k ИЛИ Wordstat 1000–5000 ИЛИ действующий сервис в РФ ИЛИ (HH>50 + 2 доп. сигнала) |
| **LOW** | Только если ВСЕ 4+ источника слабые |

**ЗАПРЕЩЕНО** ставить LOW если:
- Найден действующий сервис/бот с 1k+ пользователей
- HH показывает 200+ вакансий по смежной специальности
- Западный аналог имеет доказанный ARR > $1M

### LTV Gate — проверить ВСЕ сценарии

```
a) B2C подписка:  аудитория × конверсия 0.5-2% × месячный чек
b) B2B корпоративный: 10 клиентов × средний чек руб/мес
c) White-label/API: платформы × подписка
d) Hybrid: B2C + B2B комбинация

PASS = хотя бы один сценарий достигает 500k руб/мес с реалистичными допущениями
```

### Формат 02-demand.md

```markdown
# 02-demand
## Источники с числами
| Источник | Запрос | Результат |
|----------|--------|-----------|
| Wordstat/Trends | "..." | X/100 или X/мес |
| HH.ru | "..." | X вакансий |
| Avito | "..." | X объявлений |
| Telegram | "..." | X подписчиков |
## Composite Demand: HIGH / MEDIUM / LOW
Обоснование: [почему именно этот уровень]
## LTV Gate: PASS / FAIL
Расчёт по сценариям: [...]
## Западные аналоги
[если GEO-EXPAND]

[economics или decision: reject]
```

## 8. Program 5 — Unit Economics

**Cron:** `models-economics` (каждые 2 часа)  
**Trigger:** кейс с `02-demand.md`, без `04-economics.md`  
**Задача:** рассчитать реальную юнит-экономику в рублях.

### Метрики для расчёта

| Метрика | Формула | Порог |
|---------|---------|-------|
| **CAC** | Стоимость канала / конверсия | — |
| **LTV** | Чек × частота × срок (руб/мес) | > 500 000 руб/мес |
| **GM%** | (Выручка − Прямые затраты) / Выручка | > 40% |
| **Break-even** | Месяц выхода в плюс | < 12 мес |
| **Hourly rate** | Выручка / трудозатраты | — |

### Правило немедленного отклонения

```
ЕСЛИ LTV < 500 000 руб/мес → pipeline/rejected/
  причина: "LTV Gate не пройден: LTV = X руб/мес, порог = 500 000 руб/мес"

ЕСЛИ данных для расчёта нет → pipeline/rejected/
  причина: "Экономика не подтверждена: нет данных для расчёта CAC/LTV"
```

### Формат 04-economics.md

```markdown
# 04-economics
## Источник сигнала [из 00-brief.md]
## Данные спроса (из 02-demand.md)
## Бизнес-процесс (шаг за шагом)
| Шаг | Описание         | Время | Стоимость (руб) |
|-----|-----------------|-------|-----------------|
| 1   | ...             | X ч   | Y руб           |
## Расчёт CAC (в рублях)
## Расчёт LTV (в рублях)
LTV = [Чек × Частота × Срок] = X руб/мес
## LTV Gate
LTV = X руб/мес → PASS ✅ / FAIL ❌
## Валовая маржа: X%
## Break-even: Месяц X
## Hourly rate: X руб/час
## Итоговый вывод
## Источники
```

## 9. Program 6 — Financial Model + Critic Analysis

**Cron:** `models-finance` (каждые 2 часа)  
**Trigger:** кейс с `04-economics.md`, без `05-critic.md`  
**Задача:** P&L модель на 12 месяцев + критический анализ.

### Три сценария P&L

```
BASE:        | Месяц | Клиентов | Выручка | Затраты | Прибыль |
OPTIMISTIC:  ключевые отличия + итог на мес 12
PESSIMISTIC: ключевые отличия + итог на мес 12
```

### Критический анализ

- **Главные риски**: 3+ пункта с вероятностью % и влиянием в руб/мес
- **Конкурентная среда**: реальные конкуренты с ценами в рублях
- **Kill conditions**: что конкретно убьёт модель
- **What keeps it alive**: конкретные защитные факторы

## 10. Program 7 — Critic & Verdict

**Cron:** `models-critic` (каждые 2 часа)  
**Trigger:** кейс с `05-critic.md`, без `06-verdict.md`  
**Задача:** финальный вердикт 0–100 + отправка в Telegram.

### Scoring Criteria (0–100)

#### 1. Реальный спрос (0–20 баллов)

| Баллы | Критерий |
|-------|---------|
| 0–4 | Все источники LOW, данных нет |
| 5–9 | HH < 50 вакансий, нет TG, нет конкурентов |
| 10–14 | MEDIUM: HH > 50 ИЛИ TG 5k–50k ИЛИ конкурент в РФ |
| 15–17 | HIGH: Wordstat > 1000/мес ИЛИ Google Trends > 30 ИЛИ TG 50k+ |
| 18–20 | PROVEN: Wordstat > 5000 ИЛИ TG 150k+ ИЛИ западный аналог $10M ARR |
| +2 бонус | Действующий бот/сервис в РФ с 10k+ пользователей |

#### 2. Юнит-экономика (0–25 баллов)

| Баллы | Критерий |
|-------|---------|
| 0–5 | Нет расчётов или только гипотезы |
| 6–10 | LTV < 500k руб/мес (почти всегда REJECTED) |
| 11–15 | LTV 500k–1M, GM < 40% ИЛИ CAC > 12 мес окупаемости |
| 16–20 | LTV 500k–1M, GM 40–60%, CAC < 6 мес |
| 21–23 | LTV 1–3M руб/мес, GM > 50%, CAC < 3 мес |
| 24–25 | LTV > 3M руб/мес, GM > 60%, CAC < 2 мес |
| **Обязательно** | LTV < 500k → максимум 10/25 |

#### 3. Масштабируемость (0–20 баллов)

| Баллы | Критерий |
|-------|---------|
| 0–4 | Полностью ручная работа |
| 5–9 | Базовая систематизация, > 80% ручная |
| 10–14 | Часть автоматизирована; клиент +1 < 50% ресурсов |
| 15–17 | Productized: клиенты 2× → costs 1.3×, команда < 10 чел при 1M руб/мес |
| 18–20 | AI-native: клиенты 10× → costs 2×, SaaS/API, CAC падает с ростом |
| +2 бонус | Сервис доставляется как бот/API без ручного участия |

#### 4. Конкурентное позиционирование (0–15 баллов)

| Баллы | Критерий |
|-------|---------|
| 0–4 | Нет дифференциации |
| 5–9 | Есть уникальность, легко скопировать |
| 10–12 | Сильная ниша, сложнее скопировать |
| 13–15 | Очевидный лидер ниши или первый в сегменте в РФ |

#### 5. Защищаемость и барьеры (0–10 баллов)

| Баллы | Критерий |
|-------|---------|
| 0–3 | Нет барьеров |
| 4–6 | Умеренные (данные, интеграции, сеть) |
| 7–10 | Высокие (проприетарные данные, регулятор, патенты) |

#### 6. Качество данных (0–10 баллов)

| Баллы | Критерий |
|-------|---------|
| 0–3 | Общие слова без чисел |
| 4–6 | Часть данных с источниками |
| 7–9 | Большинство цифр со ссылками |
| 10 | Все данные верифицированы: Wordstat/Trends + HH + Avito + расчёты |

### LTV Upside Calculator (для REJECTED и NEAR-PASS)

```
1. GAP = 70 − текущий_score
2. Для каждого критерия: ПОТЕНЦИАЛ = максимум − текущий_балл
3. Отсортировать по потенциалу, взять минимальный набор покрывающий GAP
4. Для "Юнит-экономика" — ОБЯЗАТЕЛЬНО рассчитать LTV цель:
   - Если LTV < 500k: LTV_цель = 500 000 руб/мес + рычаг (цена / объём / retention)
   - Конкретно: "поднять цену с 100k до 150k руб/мес при том же числе клиентов"
5. НИКОГДА не писать "улучшить маркетинг" — только конкретные числа
```
